&gt;&gt;&gt; This is for a yet-to-exist system, so the existing data characteristics
&gt;&gt;&gt; are not that important.
&gt;&gt;&gt;
&gt;&gt;&gt; The volume of data would be something like : average 10 events per second
&gt;&gt;&gt; per source meaning about 320 million events per source, for tens of
&gt;&gt;&gt; thousands of sources, potentially hundreds of thousands.
&gt;&gt;&gt;
&gt;&gt;&gt; Data retention policy would be in the range of years, probably 5 years.
&gt;&gt;&gt;
&gt;&gt;&gt; Most of the above-mentioned are averages, some sources might be sampled
&gt;&gt;&gt; even hundreds of times per second. There is also a layer of creating
&gt;&gt;&gt; aggregates for "regressive granularity" (a la RRD) but it's a bit less of a
&gt;&gt;&gt; concern (i.e. the same strategy I'm describing could be used for storing the
&gt;&gt;&gt; aggregates.)
&gt;&gt;&gt;
&gt;&gt;&gt; The strategy I've described tries to make the most common query (time
&gt;&gt;&gt; range per source with a max number of elements) predictable and as
&gt;&gt;&gt; performant as possible. I.e. for any range I know at most three batches need
&gt;&gt;&gt; to be read from Riak (or equivalent) so I can say that, if reading a batch
&gt;&gt;&gt; takes 20 ms and the initial query takes 10 ms I can predictably respond to
&gt;&gt;&gt; most such requests under 100 ms.
&gt;&gt;&gt;
&gt;&gt;&gt; So as long as I can benchmark individual aspects of the strategy I hope
&gt;&gt;&gt; to a predictable query cost and an idea of how to grow the system.
&gt;&gt;&gt;
&gt;&gt;&gt; As for the read to write ration I don't have an exact estimate (the
&gt;&gt;&gt; system will be generic and consumption applications will be built on top of
&gt;&gt;&gt; it) but the system is expected to be a lot more write intensive than read
&gt;&gt;&gt; intensive. Most data might go completely unused, some data might be rather
&gt;&gt;&gt; "hot" so additional caching might be implemented later but I'm trying to
&gt;&gt;&gt; design the underlying system so at least some performance axioms are
&gt;&gt;&gt; computable.
&gt;&gt;&gt;
&gt;&gt;&gt; Does this clarify or confuses further?
&gt;&gt;&gt;
&gt;&gt;&gt; Regards,

&gt;&gt;&gt; On Mon, Aug 8, 2011 at 3:32 PM, Jeremiah Peschka &lt;

&gt;&gt;&gt;
&gt;&gt;&gt;&gt; It sounds like a potentially interesting use case.
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; The questions that immediately enter my head are:
&gt;&gt;&gt;&gt; \\* How much data do you currently have?
&gt;&gt;&gt;&gt; \\* How much data do you plan to have?
&gt;&gt;&gt;&gt; \\* Do you have a data retention policy? If so, what is it? How do you
&gt;&gt;&gt;&gt; plan to implement it?
&gt;&gt;&gt;&gt; \\* What's the anticipated rate of growth per day? Week? Year?
&gt;&gt;&gt;&gt; \\* What type of queries will you have? Is it a fixed set of queries? Is
&gt;&gt;&gt;&gt; it a decision support system?
&gt;&gt;&gt;&gt; \\* What does your read to write ratio look like?
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; Your plan to support Riak with a hybrid system isn't that out of whack;
&gt;&gt;&gt;&gt; it's very doable.
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; You can certainly do the type of querying you've described through
&gt;&gt;&gt;&gt; careful choice of key names, sorting in memory, and only using the first N
&gt;&gt;&gt;&gt; data points in a given Map Reduce query result. The main reason to not
&gt;&gt;&gt;&gt; perform range queries in Riak is that they'll result in full key space 
&gt;&gt;&gt;&gt; scans
&gt;&gt;&gt;&gt; across the Riak cluster. If you're using bitcask as your backend then it's
&gt;&gt;&gt;&gt; an in memory scan, otherwise you're doing a much more costly scan from 
&gt;&gt;&gt;&gt; disk.
&gt;&gt;&gt;&gt; And, since key names are hashed as they are partitioned across the cluster,
&gt;&gt;&gt;&gt; you're not going to get the benefit of sequential disk scan performance 
&gt;&gt;&gt;&gt; like
&gt;&gt;&gt;&gt; you might get with a traditional database.
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; The only thing that worries me is the phrase "should grow more than what
&gt;&gt;&gt;&gt; a 'vanilla' RDBMS would support". Are you thinking 1TB? 10TB? 50TB? 500TB?
&gt;&gt;&gt;&gt; I'm trying to get a handle on what size and performance characteristics
&gt;&gt;&gt;&gt; you're looking for before diving into how to look at your system vs. saying
&gt;&gt;&gt;&gt; "Hell if I know, does someone else on the list have a good idea?"

&gt;&gt;&gt;&gt; &gt; Hello Riak enthusiasts,
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; I am trying to design a solution for storing time series data coming
&gt;&gt;&gt;&gt; from a very large number of potential high-frequency sources.
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; I thought Riak could be of help, though based on what I read about it
&gt;&gt;&gt;&gt; I can't use it without some other layer on top of it.
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; The problem is I need to be able to do range queries over this data,
&gt;&gt;&gt;&gt; by the source. Hence, I want to be able to say "give me the N first data
&gt;&gt;&gt;&gt; points for source S between time T1 and time T2."
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; I need to store this data for a rather long time, and the expected
&gt;&gt;&gt;&gt; volume should grow more than what a "vanilla" RDBMS would support.
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; Another thing to note is that I can restrict the number of data points
&gt;&gt;&gt;&gt; to be returned by a query, so no query would return more than MaxN data
&gt;&gt;&gt;&gt; points.
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; I thought about doing this the following way:
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; 1. bundle date time series in batches of MaxN, to ensure that any
&gt;&gt;&gt;&gt; query would require reading at most two batches. The batches would be store
&gt;&gt;&gt;&gt; inside Riak.
&gt;&gt;&gt;&gt; &gt; 2. Store the start-time, end-time, size and Riak batch ID in a MySQL
&gt;&gt;&gt;&gt; (or PostgreSQL) DB.
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; My thinking is such a strategy would allow me to persist data in Riak
&gt;&gt;&gt;&gt; and linearly grow with the data, and the index would be kept in a RDBM for
&gt;&gt;&gt;&gt; fast range queries.
&gt;&gt;&gt;&gt; &gt;
&gt;&gt;&gt;&gt; &gt; Does it sound sensible to use Riak this way? Does this make you
&gt;&gt;&gt;&gt; laugh/cry/shake your head in disbelief? Am I overlooking something from 
&gt;&gt;&gt;&gt; Riak
&gt;&gt;&gt;&gt; which would make all this much better?
