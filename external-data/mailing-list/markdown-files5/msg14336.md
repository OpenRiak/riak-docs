In my testing with large but smaller binaries (median 40k), I found
that the settings gave a noticeable bump (8000 -&gt; 10000 ops/s) , but
only so far as the disk could keep up (and the disk cache, of course).
Typically, for larger objects, you're going to be disk limited most of
the time. Remember that riak is basically doing a bunch of random,
mid-sized (to the disk, at least) reads and writes here, so disk
limitations are going to make it really hard to get near to your
disk's theoretical maximums.

&gt;&gt; Hi Earl,
&gt;&gt;
&gt;&gt; There are some known internode bottlenecks in riak 1.4.x. We've
&gt;&gt; addressed some of them in 2.0, but others likely remain. If you're
&gt;&gt; willing to run some code at the console, running the following at the
&gt;&gt; console (from `riak attach`) should tell you whether or not the 2.0
&gt;&gt; changes are likely to help you. I am not sure when 2.0 ready versions
&gt;&gt; of CS are slated for, however.
&gt;&gt;
&gt;&gt; -----
&gt;&gt; [inet:setopts(Port, [{sndbuf, 393216}, {recbuf, 786432}])
&gt;&gt; || {\\_Node, Port} &lt;- erlang:system\\_info(dist\\_ctrl)].
&gt;&gt;
&gt;&gt; or to run this on all nodes (which you'll have to do to see if it helps):
&gt;&gt;
&gt;&gt; FF = fun() -&gt;
&gt;&gt; [inet:setopts(Port, [{sndbuf, 393216}, {recbuf, 786432}])
&gt;&gt; || {\\_Node, Port} &lt;- erlang:system\\_info(dist\\_ctrl)]
&gt;&gt; end.
&gt;&gt; rpc:multicall(erlang, apply, [FF, []]).
&gt;&gt;
&gt;&gt; You should not run any of this on production machines without
&gt;&gt; extensive testing first. Also if you have huge objects, like in a CS
&gt;&gt; cluster, it may help to increase the buffer sizes somewhat.
&gt;&gt;
&gt;&gt; Note that increasing +zdbbl in your vm.args can also help somewhat, if
&gt;&gt; it isn't already prohibitively large.
&gt;&gt;
&gt;&gt; Hope that this helps. Let us know what you find.
