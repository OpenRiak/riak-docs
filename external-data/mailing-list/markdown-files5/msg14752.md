Yes, it works automatically. All included datatypes are available inside maps.

&gt;
&gt; 2. if we use "number\\_counter" instead of "number\\_i" does Search/SOLR
&gt; understand is an integer? in case you want to do a range... as somewhere in
&gt; the docs I read that better to use "\\_s" for strings, "\\_b" for binary, "\\_i"
&gt; for integers, etc... so SOLR knows how to treat the data... I believe there
&gt; will be no strange behaviours for having "\\_register" instead of "\\_s" and
&gt; "\\_counter" instead of "\\_i", right?

The default Solr schema that ships with Riak accounts for these
datatypes automatically and uses the appropriate index field type:
https://github.com/basho/yokozuna/blob/develop/priv/default\\_schema.xml#L96-L104

If you write your own schema, you will want to include or change the
schema fields appropriately.

&gt;&gt; Alex,
&gt;&gt;
&gt;&gt; Assuming you've already made your bucket-type with "map" as the
&gt;&gt; datatype, then "bucket.new()" will return you a Map instead of a
&gt;&gt; RiakObject. Translating your example above:
&gt;&gt;
&gt;&gt; key = bucket.new('lionelmessi')
&gt;&gt; key.registers['name'].assign('Messi')
&gt;&gt; key.registers['team'].assign('Barcelona')
&gt;&gt; key.counters['number'].increment(10)
&gt;&gt; key.store()
&gt;&gt;
&gt;&gt; Note that because Maps are based on mutation operations and not
&gt;&gt; replacing the value with new ones, you can later do this without
&gt;&gt; setting the entire value:
&gt;&gt;
&gt;&gt; key.counters['number'].increment(1)
&gt;&gt; key.store()
&gt;&gt;
&gt;&gt; This will also change your searches, however, in that the fields will
&gt;&gt; be suffixed with the embedded type you are using:
&gt;&gt;
&gt;&gt; r = client.fulltext\\_search('ix\\_footballers', 'team\\_register:Barcelona')
&gt;&gt;
&gt;&gt; Hope that helps!
