&gt; On Tue, Oct 27, 2015 at 4:06 PM Jon Meredith  wrote:
&gt;
&gt;&gt; Hi,
&gt;&gt;
&gt;&gt; Handoff problems without obvious disk issues can be due to the database
&gt;&gt; containing large objects. Do you frequently update objects in CS, and if
&gt;&gt; so have you had garbage collection running?
&gt;&gt;
&gt;&gt; The timeout is happening on the receiver side after not receiving any tcp
&gt;&gt; data for handoff\\_receive\\_timeout \\*milli\\*seconds. I know you said you
&gt;&gt; increased it, but not how high. I would bump that up to 300000 to give the
&gt;&gt; sender a chance to read larger objects off disk.
&gt;&gt;
&gt;&gt; To check if the sender is transmitting, on the source node you could run
&gt;&gt; redbug:start("riak\\_core\\_handoff\\_sender:visit\\_item", [{arity,
&gt;&gt; true},{print\\_file,"/tmp/visit\\_item.log"},{time, 3600000},{msgs, 1000000}]).
&gt;&gt;
&gt;&gt; That file should fill fairly fast with an entry for every object the
&gt;&gt; sender tries to transmit.
&gt;&gt;
&gt;&gt; There's a long shot it could be network misconfiguration. Run this from
&gt;&gt; the source node having problems
&gt;&gt;
&gt;&gt; rpc:multicall(erlang, apply, [fun() -&gt; TargetNode = node(), [\\_Name,Host]
&gt;&gt; = string:tokens(atom\\_to\\_list(TargetNode), "@"), {ok, Port} =
&gt;&gt; riak\\_core\\_gen\\_server:call({riak\\_core\\_handoff\\_listener, TargetNode},
&gt;&gt; handoff\\_port), HandoffIP = riak\\_core\\_handoff\\_listener:get\\_handoff\\_ip(),
&gt;&gt; TNHandoffIP = case HandoffIP of error -&gt; Host; {ok, "0.0.0.0"} -&gt; Host;
&gt;&gt; {ok, Other} -&gt; Other end, {node(), HandoffIP, TNHandoffIP,
&gt;&gt; inet:gethostbyname(TNHandoffIP), Port} end, []]).
&gt;&gt;
&gt;&gt; and it will print out a a list of remote nodes and IP addresses (and
&gt;&gt; hopefully an empty list of failed nodes)
&gt;&gt;
&gt;&gt; {[{'dev1@127.0.0.1', &lt;---- node name
&gt;&gt;
&gt;&gt; {ok,"0.0.0.0"}, &lt;---- handoff ip address configured in
&gt;&gt; app.config
&gt;&gt;
&gt;&gt; "127.0.0.1", &lt;---- hostname passed to socket open
&gt;&gt;
&gt;&gt; {ok,{hostent,"127.0.0.1",[],inet,4,[{127,0,0,1}]}}, &lt;--- DNS entry for
&gt;&gt; hostname
&gt;&gt;
&gt;&gt; 10019}], &lt;---- handoff port
&gt;&gt;
&gt;&gt; []} &lt;--- empty list of errors
&gt;&gt;
&gt;&gt; Good luck, Jon.

&gt;&gt;&gt;&gt; Hi,
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; I suspect your {error,enotconn} messages are unrelated - that's likely
&gt;&gt;&gt;&gt; to be caused by an HTTP client closing the connection while Riak looks up
&gt;&gt;&gt;&gt; some networking information about the requestor.
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; The max\\_concurrency message you are seeing is related to the handoff
&gt;&gt;&gt;&gt; transfer limit - it should be labelled as informational. When a node has
&gt;&gt;&gt;&gt; data to handoff it starts the handoff sender process and if there are
&gt;&gt;&gt;&gt; either too many local handoff processes or too many on the remote side it
&gt;&gt;&gt;&gt; exits with max\\_concurrency. You could increase with riak-admin
&gt;&gt;&gt;&gt; transfer-limit but that probably won't help if you're timing out.
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; As you're using the multi-backend you're transferring data from bitcask
&gt;&gt;&gt;&gt; and leveldb. The next place I would look is in the leveldb LOG files to
&gt;&gt;&gt;&gt; see if there are any leveldb vnodes that are having problems that's
&gt;&gt;&gt;&gt; preventing repair.
