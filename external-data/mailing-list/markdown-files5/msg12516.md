&gt;&gt; &gt; As a side question, what is the best way to delete such an object i.e.
&gt;&gt; &gt; once
&gt;&gt; &gt; I know one of the siblings has 'deleted' flag true because I fetched it?
&gt;&gt; &gt; Should I just use DomainBucket.delete(key) without providing any vclock?
&gt;&gt; &gt; Would it wipe it from Riak or create yet another sibling?
&gt;&gt;
&gt;&gt; You should always use vclocks when possible, which in the case it is.
&gt;&gt; There are additional issues around doing the delete without a vclock
&gt;&gt; and if there's concurrently a store operation occurring.
&gt;&gt;
&gt;&gt; Ideally you should look at why you're getting that tombstone sibling.
&gt;&gt; If it's simply a case of high write concurrency and you're using
&gt;&gt; vclocks with your writes, then there's not much you can do except
&gt;&gt; resolve it later (without changing how you're using the DB)... but
&gt;&gt; usually these things are caused by writes without a vclock.
