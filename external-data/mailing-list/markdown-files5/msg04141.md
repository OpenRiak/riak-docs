On Jul 29, 2011, at 1:37 AM, Antonio Rohman Fernandez wrote:

&gt; MapReduce ( or a simply Map ) gets really slow when database has a 
&gt; significant amount of data ( or distributed over several servers ). Get 
&gt; instead is always faster as Riak doesn't have to search for the key ( you 
&gt; tell Riak exactly where to GET the data in your url )
&gt; 
&gt; Rohman
&gt; 
&gt; On Thu, 28 Jul 2011 23:43:06 +0400, m...@mawhrin.net wrote:
&gt; 
&gt;&gt; Hi,
&gt;&gt; 
&gt;&gt; (I looked at various places for the information, however I could not
&gt;&gt; find anything that would answer the question. It's not completely ruled
&gt;&gt; out that not all places were checked though :))
&gt;&gt; 
&gt;&gt; I use PB erlang interface to access the database. Given a bucket name
&gt;&gt; and a key, the value can easily be extracted using:
&gt;&gt; 
&gt;&gt; {ok, Object} = riakc\\_pb\\_socket:get(Conn, Bucket, Key),
&gt;&gt; Value = riakc\\_obj:get\\_value(Object)
&gt;&gt; 
&gt;&gt; Alternatively, a mapred (actually, just map) request could be issued:
&gt;&gt; 
&gt;&gt; {ok, [{\\_, Value}]} = riakc\\_pb\\_socket:mapred(Conn, [
&gt;&gt; {Bucket, Key}
&gt;&gt; ], [
&gt;&gt; {map, {modfun, riak\\_kv, map\\_object\\_value}, none, true}
&gt;&gt; ])
&gt;&gt; 
&gt;&gt; I would expect that the result is the same while in the second case, the
&gt;&gt; amount of data transferred to the client is smaller (which might be good
&gt;&gt; for certain situations).
&gt;&gt; 
&gt;&gt; So the [open] question is: are there any reasons for using the first
&gt;&gt; approach over the second?
