Ingo -

Unfortunately once you've got a sibling tombstone, things get a bit
tricky. It's not going away until you resolve them which when using
the JSONConverter in the Java client, you can't. Oddly enough, this is
the first time anyone has hit this.

I've got a couple ideas on how to address this properly but I need to
look at some things first.

In the meantime, what I'd suggest as a workaround is to copy and paste
the source for the JSONConverter into your own Converter that
you'll pass to the StoreObject and modify it to return null:

https://github.com/basho/riak-java-client/blob/master/src/main/java/com/basho/riak/client/convert/JSONConverter.java#L141

Have it check to see if riakObject.getValue() returns null and if it
does ... return null. You'll also need to modify your ConflictResolver
to check for null as it iterates through the list of your POJOs that
gets passed to it and act accordingly. If there's only a tombstone,
just return null ... which means you will also need to modify your
Mutation to handle a null being passed to it in the case of there
only being a tombstone.

In the end this may well be what I do but I think I have a slightly
more elegant solution that I want to look into.

I've got an errand I need to run this morning, but I'll get to work on
this as soon as I get back.

Thanks, and sorry for the trouble.
- Brian Roach

&gt;&gt; Ingo -

&gt;&gt; Riak is returning an object with no contents (which ends up being an
&gt;&gt; empty String passed to Jackson).
&gt;&gt;
&gt;&gt; Unless you've somehow mangled the data yourself (which sounds unlikely
&gt;&gt; given the bit about the 404 from the command line; more on that in a
&gt;&gt; bit) what's happening is that you're encountering a tombstone; an
&gt;&gt; object that has been deleted via a delete operation but hasn't been
&gt;&gt; removed yet. This causes an "empty" object to be returned (the
&gt;&gt; tombstone) and causes Jackson to puke (HTTP will actually return this
&gt;&gt; as a 404 but if you look there's still a X-Riak-Vclock: header with a
&gt;&gt; vclock).
&gt;&gt;
&gt;&gt; Probably the best description of how this works in Riak is a post by
&gt;&gt; Jon Meredith which can be found here:
&gt;&gt;
&gt;&gt; http://lists.basho.com/pipermail/riak-users\\_lists.basho.com/2011-October/006048.html
&gt;&gt;
&gt;&gt; Unfortunately this is something the Java client doesn't know what to
&gt;&gt; do with when using the default JSONConverter and your own POJOs. And
&gt;&gt; it's not as simple as "just return null" because of a case where a
&gt;&gt; tombstone could actually be a sibling and the client then needs to
&gt;&gt; resolve the conflict which is the next step in the process. It's
&gt;&gt; something I'm going to have to think about.
&gt;&gt;
&gt;&gt; As you've discovered, when the tombstone isn't a sibling simply
&gt;&gt; retrying will often work because by then the delete has fully
&gt;&gt; completed and the tombstones have been removed from the Riak nodes.
&gt;&gt;
&gt;&gt; Is there a reason you're rapidly doing a delete then a store (which
&gt;&gt; triggers that fetch)?
