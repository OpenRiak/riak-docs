&gt; &gt; There is now a second update available for 2.9.0:
&gt; &gt; https://github.com/basho/riak/tree/riak-2.9.0p2.
&gt; &gt;
&gt; &gt; This patch, like the patch before, resolves a memory management issue in
&gt; &gt; leveled, which this time could be triggered by sending many large objects
&gt; &gt; in a short period of time. The underlying problem is described a bit
&gt; &gt; further here https://github.com/martinsumner/leveled/issues/285, and is
&gt; &gt; resolved by leveled working more sympathetically with the beam binary
&gt; &gt; memory management.
&gt; &gt;
&gt; &gt; Switching to the patched version is not urgent unless you are using the
&gt; &gt; leveled backend, and may send a large number of large objects in a burst.
&gt; &gt;
&gt; &gt; Updated packages are available (thanks to Nick Adams at TI Tokyo) -
&gt; &gt; https://files.tiot.jp/riak/kv/2.9/2.9.0p2/
&gt; &gt;
&gt; &gt; Thanks again to the testing team at the NHS Spine project, Aaron Gibbon
&gt; &gt; (BJSS) and Ramen Sen, who discovered the problem. The issue was
&gt; discovered
&gt; &gt; in a handoff scenario where there were a tens of thousands of 2MB objects
&gt; &gt; stored in a portion of the keyspace at the end of the handoff - which led
&gt; &gt; to memory issues until either more PUTs were received (to force a persist
&gt; &gt; to disk) or a restart occurred..
&gt; &gt;

&gt; &gt; Bryan,
&gt; &gt;
&gt; &gt; We saw that Riak was using much more memory than was expected at the end
&gt; of
&gt; &gt; the handoffs. Using `riak-admin top` we could see that this wasn't
&gt; process
&gt; &gt; memory, but binaries. Firstly did some work via attach looping over
&gt; &gt; processes and running GC to confirm that this wasn't a failure to collect
&gt; &gt; garbage - the references to memory were real. Then did a bit of work in
&gt; &gt; attach writing some functions to analyse process\\_info/2 for each process
&gt; &gt; (looking at binary and memory), and discovered that there were penciller
&gt; &gt; processes that had lots of references to lots of large binaries (and this
&gt; &gt; accounted for all the unexpected memory use), and where the penciller was
&gt; &gt; the only process with a reference to the binary. This made no sense
&gt; &gt; initially as the penciller should only have small binaries (metadata).
&gt; &gt; Then looked at the running state of the penciller processes and could see
&gt; &gt; no large binaries in the state, but could see that a lot of the active
&gt; keys
&gt; &gt; in the penciller were keys that were known to have large object values
&gt; (but
&gt; &gt; small amounts of metadata) - and that the size of the object values were
&gt; &gt; the same as the size of the binary references found on the penciller
&gt; &gt; process via process\\_info/2..
&gt; &gt;
&gt; &gt; I then recalled the first part of this:
&gt; &gt;
&gt; https://dieswaytoofast.blogspot.com/2012/12/erlang-binaries-and-garbage-collection.html
&gt; .
&gt; &gt; It was obvious that in extracting the metadata the beam was naturally
&gt; &gt; retaining a reference to the whole binary, as long as the sub-binary was
&gt; &gt; retained by the a process (the Penciller). Forcing a binary copy
&gt; resolved
&gt; &gt; this referencing issue. It was nice that the same tools used to detect
&gt; the
&gt; &gt; issue, made it quite easy to write a test to confirm resolution -
&gt; &gt;
&gt; https://github.com/martinsumner/leveled/blob/master/test/end\\_to\\_end/riak\\_SUITE.erl#L1214-L1239
&gt; &gt; .
&gt; &gt;
&gt; &gt; The memory leak section of Fred Herbert's
&gt; http://www.erlang-in-anger.com/ is
&gt; &gt; great reading for helping with these types of issues.
&gt; &gt;
&gt; &gt; Thanks
&gt; &gt;

&gt; the active keys in the penciller were keys that were known to have large
&gt; object values (but small amounts of metadata) - and that the size of the
&gt; object values were the same as the size of the binary references found on
&gt; the penciller process via process\\_info/2..
&gt; &gt;&gt;
&gt; &gt;&gt; I then recalled the first part of this:
&gt; https://dieswaytoofast.blogspot.com/2012/12/erlang-binaries-and-garbage-collection.html
&gt; &lt;
&gt; https://dieswaytoofast.blogspot.com/2012/12/erlang-binaries-and-garbage-collection.html&gt;.
&gt; It was obvious that in extracting the metadata the beam was naturally
&gt; retaining a reference to the whole binary, as long as the sub-binary was
&gt; retained by the a process (the Penciller). Forcing a binary copy resolved
&gt; this referencing issue. It was nice that the same tools used to detect the
&gt; issue, made it quite easy to write a test to confirm resolution -
&gt; https://github.com/martinsumner/leveled/blob/master/test/end\\_to\\_end/riak\\_SUITE.erl#L1214-L1239
&gt; &lt;
&gt; https://github.com/martinsumner/leveled/blob/master/test/end\\_to\\_end/riak\\_SUITE.erl#L1214-L1239
&gt; &gt;.
&gt; &gt;&gt;
&gt; &gt;&gt; The memory leak section of Fred Herbert's
&gt; http://www.erlang-in-anger.com/  is
&gt; great reading for helping with these types of issues.

&gt; &gt;&gt; This patch, like the patch before, resolves a memory management issue
&gt; in leveled, which this time could be triggered by sending many large

&gt; https://github.com/martinsumner/leveled/issues/285&gt;, and is resolved by
&gt; leveled working more sympathetically with the beam binary memory
&gt; management.
&gt; &gt;&gt;
&gt; &gt;&gt; Switching to the patched version is not urgent unless you are using the
&gt; leveled backend, and may send a large number of large objects in a burst.
