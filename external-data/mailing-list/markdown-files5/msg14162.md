&gt; The 2.0 solution to this is to create a bucket type

&gt; buckets of that type created later will inherit that setting, so you
&gt; should be able to create as many of them as you like.

&gt; &gt;&gt;
&gt; &gt;&gt; I suspect that, given the large heap messages, you're seeing the known
&gt; &gt;&gt; issues where when a custom bucket is created and the moving that
&gt; &gt;&gt; metadata around the ring takes increasingly long. It isn't
&gt; &gt;&gt; recommended to create a large number of custom buckets at the moment.
&gt; &gt;&gt;
&gt; &gt;&gt; What the process likely looks like is this:
&gt; &gt;&gt;
&gt; &gt;&gt; 1. you do the set.
&gt; &gt;&gt; 2. bucket metadata starts being gossip around the ring.
&gt; &gt;&gt; 3. you do the put. it succeeds, but on some nodes, the metadata
&gt; &gt;&gt; hasn't been committed, so it's put into the default backend.
&gt; &gt;&gt; 4. gossip finishes.
&gt; &gt;&gt; 5. you do the get. it fails because now the bucket for this backend
&gt; &gt;&gt; has made 2 replicas unreachable.
&gt; &gt;&gt; 6. read repair happens, repopulating the 2 missing replicas.
&gt; &gt;&gt; 7. you re-get and it works.
