comparisons on the wiki, but this is a good differentiator re:

&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; Riak's usage of memory primarily depends on the backend you choose.
&gt;&gt;&gt;&gt;  Innostore, for example, has a configurable buffer pool (cache) which can
&gt;&gt;&gt;&gt; help you limit the memory footprint.  Bitcask, our most recently released
&gt;&gt;&gt;&gt; backend, keeps only a hash mapping keys to file/offset in memory (with the
&gt;&gt;&gt;&gt; value only on disk), so you could store thousands of keys per node without
&gt;&gt;&gt;&gt; noticing much.
&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt; Outside of the backend, Riak generally only needs enough RAM to have
&gt;&gt;&gt;&gt; copies of objects in transit -- either being written or read.  If you see 
&gt;&gt;&gt;&gt; it
&gt;&gt;&gt;&gt; using too much RAM, there are some flags to the Erlang VM that can be
&gt;&gt;&gt;&gt; tweaked.
