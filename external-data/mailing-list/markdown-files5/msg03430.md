&gt;&gt; ),
&gt;&gt; it is cached until you either refresh it manually by passing :reload =&gt; true
&gt;&gt; or a block (for streaming key lists). This was the compromise reached in
&gt;&gt; that pull-request.
&gt;&gt;
&gt;&gt; All of this caching discussion glosses over the fact that you \\*should
&gt;&gt; not list keys\\* in any real application. It really begs the question --
&gt;&gt; how often do you list keys in Redis, or memcached? I suspect that generally
&gt;&gt; you don't. This isn't a relational database. (Also, how often do you
&gt;&gt; actually do a full-table scan in MySQL? You don't if you're sane -- you use
&gt;&gt; an index, or even LIMIT + OFFSET.)
&gt;&gt;
&gt;&gt; I'm tempted to remove Document::all and make Bucket#keys harder to access,
&gt;&gt; but the balance between discouraging bad behavior and exposing available
&gt;&gt; functionality is a hard one to strike. I don't want new developers to
&gt;&gt; immediately use list-keys and then be discouraged from using Riak because
&gt;&gt; it's slow; on the other hand, it \\*can be useful\\* in some circumstances.
&gt;&gt;
&gt;&gt;
&gt;&gt;
&gt;&gt; In those cases where it's useful, the developer should probably be
&gt;&gt; responsible enough to request the key list only once; the caching behavior
&gt;&gt; simply does this for them. I guess whether it \\*should\\* do this for them
&gt;&gt; is the issue at hand.
&gt;&gt;
&gt;&gt;
&gt;&gt; YES! Exactly! The decision to expose the functionality has been made;
&gt;&gt; questioning whether or not this should have been done is orthogonal to
&gt;&gt; whether or not the results should be cached, and the two should be
&gt;&gt; considered separately.
&gt;&gt;
&gt;&gt; Regarding the latter, the function name represents an implied promise to
&gt;&gt; the caller; my position is that the function's behavior is a substantial and
&gt;&gt; surprising deviation from that implied promise.
&gt;&gt;
&gt;&gt; Although buckets do not \\*contain\\* key/values in the riak \\*implementation\\*,
&gt;&gt; the bucket / key-value containment metaphor pervades the developer
&gt;&gt; interface, evidenced by, for example, the existence of the Riak::Bucket
&gt;&gt; class, and the structure of the URL's with which values are manipulated. In
&gt;&gt; software products that have containment metaphors, how often do we see a
&gt;&gt; function return a cached value rather than the up-to-date value, especially
&gt;&gt; for products that manage shared data?
&gt;&gt;
&gt;&gt; All that said, I'm really torn on this issue, and the same problem applies
&gt;&gt; to full-bucket MapReduce. Caveat emptor.
