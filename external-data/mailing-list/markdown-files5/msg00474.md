On May 28, 2010, at 11:51 PM, Sam Tingleff wrote:

&gt; I've filed issue #5 on the java client: RiakObject should not present
&gt; a getValueAsBytes() method
&gt; 
&gt; The real problem is that if the caller does not use stream(),
&gt; RiakClient will call getResponseBodyAsString() on the HttpMethod
&gt; object. This will of course convert from byte[] to String, which is
&gt; broken behavior for non-string data and is unexpected if the caller
&gt; uses getValueAsBytes() in RiakObject.
&gt; 
&gt; IMHO RiakObject should only hold a byte array and perform byte[] to
&gt; String conversion only on demand and when explicitly asked to do so
&gt; (if at all).
&gt; 
&gt; On Fri, May 28, 2010 at 3:29 PM, Sam Tingleff  wrote:
&gt;&gt;

&gt;&gt;

&gt;&gt;&gt; Your other alternative is to write with the Java client, then download the 
&gt;&gt;&gt; data with an external program like curl, and see if the bytes are mangled 
&gt;&gt;&gt; before they get to Riak (using a hex editor or something). You might even 
&gt;&gt;&gt; use something like Wireshark to see what is sent in the TCP packets, if you 
&gt;&gt;&gt; want to get fancy.
&gt;&gt;&gt; My suspicion is that they are being modified somewhere in the Java stack. 
&gt;&gt;&gt; Riak is completely agnostic about any binary data you give it.
