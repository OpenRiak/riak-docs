Hi Dmitri, this thread is old, but I read this part of your answer
carefully:

You can use the following strategies to prevent stale values, in increasing
&gt; order of security/preference:
&gt; 1) Use timestamps (and not pass in vector clocks/causal context). This is
&gt; ok if you're not editing objects, or you're ok with a bit of risk of stale
&gt; values.
&gt; 2) Use causal context correctly (which means, read-before-you-write -- in
&gt; fact, the Update operation in the java client does this for you, I think).
&gt; And if Riak can't determine which version is correct, it will fall back on
&gt; timestamps.
&gt; 3) Turn on siblings, for that bucket or bucket type. That way, Riak will
&gt; still try to use causal context to decide the right value. But if it can't
&gt; decide, it will store BOTH values, and give them back to you on the next
&gt; read, so that your application can decide which is the correct one.


I decided on strategy #2. What I am hoping for is some validation that the
code we use to "get", "put", and "delete" is correct in that context, or if
it could be simplified in some cases. Not we are using delete-mode
"immediate" and no duplicates.

In their shortest possible forms, here are the three methods I'd like some
feedback on (note, they're being used in production and haven't caused any
problems yet, however we have very few writes in production so the lack of
problems doesn't support the conclusion that the implementation is
correct.) Note all argument-checking, exception-handling, and logging
removed for clarity. \\*I'm mostly concerned about correct use of causal
context and response.isNotFound and response.hasValues. \\*Is there anything
I could/should have left out?

 public RiakClient(String name, com.basho.riak.client.api.RiakClient
client)
 {
 this.name = name;
 this.namespace = new Namespace(name);
 this.client = client;
 }

 public byte[] get(String key) throws ExecutionException,
InterruptedException {

 FetchValue.Response response = fetchValue(key);
 if (!response.isNotFound())
 {
 RiakObject riakObject = response.getValue(RiakObject.class);
 return riakObject.getValue().getValue();
 }
 return null;
 }

 public void put(String key, byte[] value) throws ExecutionException,
InterruptedException {

 // fetch in order to get the causal context
 FetchValue.Response response = fetchValue(key);
 RiakObject storeObject = new

RiakObject().setValue(BinaryValue.create(value)).setContentType("binary/octet-stream");
 StoreValue.Builder builder =
 new StoreValue.Builder(storeObject).withLocation(new
Location(namespace, key));
 if (response.getVectorClock() != null) {
 builder = builder.withVectorClock(response.getVectorClock());
 }
 StoreValue storeValue = builder.build();
 client.execute(storeValue);
 }

 public boolean delete(String key) throws ExecutionException,
InterruptedException {

 // fetch in order to get the causal context
 FetchValue.Response response = fetchValue(key);
 if (!response.isNotFound())
 {
 DeleteValue deleteValue = new DeleteValue.Builder(new
Location(namespace, key)).build();
 client.execute(deleteValue);
 }
 return !response.isNotFound() || !response.hasValues();
 }

&gt; Hi Vanessa,
&gt;
&gt; The thing to keep in mind about read repair is -- it happens
&gt; asynchronously on every GET, but /after/ the results are returned to the
&gt; client.
&gt;
&gt; So, when you issue a GET with r=1, the coordinating node only waits for 1
&gt; of the replicas before responding to the client with a success, and only
&gt; afterwards triggers read-repair.
&gt;
&gt; It's true that with notfound\\_ok=false, it'll wait for the first
&gt; non-missing replica before responding. But if you edit or update your
&gt; objects at all, an R=1 still gives you a risk of stale values being
&gt; returned.
&gt;
&gt; For example, say you write an object with value A. And let's say your 3
&gt; replicas now look like this:
&gt;
&gt; replica 1: A, replica 2: A, replica 3: notfound/missing
&gt;
&gt; A read with an R=1 and notfound\\_ok=false is just fine, here. (Chances are,
&gt; the notfound replica will arrive first, but the notfound\\_ok setting will
&gt; force the coordinator to wait for the first non-empty value, A, and return
&gt; it to the client. And then trigger read-repair).
&gt;
&gt; But what happens if you edit that same object, and give it a new value,
&gt; B? So, now, there's a chance that your replicas will look like this:
&gt;
&gt; replica 1: A, replica 2: B, replica 3: B.
&gt;
&gt; So now if you do a read with an R=1, there's a chance that replica 1, with
&gt; the old value of A, will arrive first, and that's the response that will be
&gt; returned to the client.
&gt;
&gt; Whereas, using R=2 eliminates that risk -- well, at least decreases it.
&gt; You still have the issue of -- how does Riak decide whether A or B is the
&gt; correct value? Are you using causal context/vclocks correctly? (That is,
&gt; reading the object before you update, to get the correct causal context?)
&gt; Or are you relying on timestamps? (This is an ok strategy, provided that
&gt; the edits are sufficiently far apart in time, and you don't have many
&gt; concurrent edits, AND you're ok with the small risk of occasionally the
&gt; timestamp being wrong). You can use the following strategies to prevent
&gt; stale values, in increasing order of security/preference:
&gt;
&gt; 1) Use timestamps (and not pass in vector clocks/causal context). This is
&gt; ok if you're not editing objects, or you're ok with a bit of risk of stale
&gt; values.
&gt;
&gt; 2) Use causal context correctly (which means, read-before-you-write -- in
&gt; fact, the Update operation in the java client does this for you, I think).
&gt; And if Riak can't determine which version is correct, it will fall back on
&gt; timestamps.
&gt;
&gt; 3) Turn on siblings, for that bucket or bucket type. That way, Riak will
&gt; still try to use causal context to decide the right value. But if it can't
&gt; decide, it will store BOTH values, and give them back to you on the next
&gt; read, so that your application can decide which is the correct one.
&gt;
&gt;

&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt; The drawbacks to load-balancing on the java client side (yes, the
&gt;&gt;&gt;&gt;&gt; cluster object) instead of a standalone load balancer like HAProxy, are 
&gt;&gt;&gt;&gt;&gt; the
&gt;&gt;&gt;&gt;&gt; following:
&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt; 1) Adding node means code changes (or at very least, config file
&gt;&gt;&gt;&gt;&gt; changes) rolled out to all your clients. Which turns out to be a pretty
&gt;&gt;&gt;&gt;&gt; serious hassle. Instead, HAProxy allows you to add or remove nodes without
&gt;&gt;&gt;&gt;&gt; changing any java code or config files.
&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt; 2) Performance. We've ran many tests to compare performance, and
&gt;&gt;&gt;&gt;&gt; client-side load balancing results in significantly lower throughput than
&gt;&gt;&gt;&gt;&gt; you'd have using haproxy (or nginx). (Specifically, you actually want to
&gt;&gt;&gt;&gt;&gt; use the 'leastconn' load balancing algorithm with HAProxy, instead of 
&gt;&gt;&gt;&gt;&gt; round
&gt;&gt;&gt;&gt;&gt; robin).
&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt; 3) The health check on the client side (so that the java load balancer
&gt;&gt;&gt;&gt;&gt; can tell when a remote node is down) is much less intelligent than a
&gt;&gt;&gt;&gt;&gt; dedicated load balancer would provide. With something like HAProxy, you
&gt;&gt;&gt;&gt;&gt; should be able to take down nodes with no ill effects for the client code.
&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt; Now, if you load balance on the client side and you take a node down,
&gt;&gt;&gt;&gt;&gt; it's not supposed to stop working completely. (I'm not sure why it's
&gt;&gt;&gt;&gt;&gt; failing for you, we can investigate, but it'll be easier to just use a 
&gt;&gt;&gt;&gt;&gt; load
&gt;&gt;&gt;&gt;&gt; balancer). It should throw an error or two, but then start working again
&gt;&gt;&gt;&gt;&gt; (on the retry).
