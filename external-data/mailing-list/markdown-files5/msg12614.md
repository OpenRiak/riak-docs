Not sure it's the greatest way (sometimes slow), but our integration
testing tool riak\\_test uses git to store a clean "devrel" and resets/cleans
the git repository at the beginning of each test. If you're curious, I can
point you to the relevant sections of code that do this.

&gt;&gt; Hi,
&gt;&gt; I'd like to hear how other people are approaching the problem of cleaning
&gt;&gt; Riak buckets up at the end of unit tests for their apps.
&gt;&gt;
&gt;&gt; The problem I have is that multiple tests may be run at once (by
&gt;&gt; different developers or different Jenkins' jobs or even just a parallelised
&gt;&gt; test suite) so I can't really run a blanket delete-all at the end of the
&gt;&gt; test suite, unless I use a randomly-named bucket each time. Yet if I do
&gt;&gt; that, I'm concerned the test suite may crash out prior to the end
&gt;&gt; sometimes, and then never delete that randomly-named bucket.
&gt;&gt;
&gt;&gt;
&gt;&gt; If secondary indexes aren't required, then the easy solution is to use a
&gt;&gt; randomly-named Bitcask bucket which has a backend configured for a fairly
&gt;&gt; short TTL.
&gt;&gt;
&gt;&gt;
&gt;&gt; Otherwise, I have wondered about creating buckets with a certain format,
&gt;&gt; perhaps "test-XXXXXX-YYYY-MM-DD", (x=random) and then a nightly cron script
&gt;&gt; can run to find all buckets timestamped from the previous day or earlier,
&gt;&gt; and remove them. I gather listing all buckets is an expensive operation
&gt;&gt; though, although it'll only be running on a testing Riak cluster.
&gt;&gt;
&gt;&gt;
&gt;&gt; So I wondered how other developers are approaching this issue?
