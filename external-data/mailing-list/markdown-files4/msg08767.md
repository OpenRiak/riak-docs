&gt; On Mon, Oct 1, 2012 at 3:14 PM, Sean Cribbs  wrote:
&gt;&gt;
&gt;&gt; In contrast to Yuri's claim, a number of groups have used Riak
&gt;&gt; successfully as a cache. Yes, it won't be as fast as Membase, but it
&gt;&gt; might be fast enough for your purposes. (This presentation by
&gt;&gt; Posterous might interest you:
&gt;&gt;
&gt;&gt; http://basho.com/blog/technical/2012/01/30/Riak-in-Production-at-Posterous-Riak-Control-Preview/)
&gt;&gt;
&gt;&gt; 1) Bitcask is going to be very fast. If your data never really exceeds
&gt;&gt; RAM, Bitcask will give you near-RAM speeds with the assurance that
&gt;&gt; your cache will also persist to disk. Normal Bitcask caveats apply
&gt;&gt; here, including the "keys-must-fit-in-RAM" one.
&gt;&gt; 2) For most persistent use-cases it is recommended to have &gt; 3 nodes,
&gt;&gt; preferably 5 or more. For a cache, the guarantees of per-key
&gt;&gt; availability are usually more lax, so perhaps &lt; 3 nodes is ok. It's
&gt;&gt; better to set up your desired configuration, run some benchmarks and
&gt;&gt; smoke-tests while stopping or killing nodes.
&gt;&gt; 3) If you are using the "memory" backend, you can configure this
&gt;&gt; behavior. If you do not configure it, the default is to use up to as
&gt;&gt; much memory as the operating system will allow it -- beyond that
&gt;&gt; limit, the Riak node will exit and leave a crash dump. If you
&gt;&gt; configure a size or TTL limit, then the memory backend will drop the
&gt;&gt; oldest item from its storage. More information on how to configure it
&gt;&gt; is here: http://wiki.basho.com/Memory.html
&gt;&gt;
&gt;&gt; I hope that helps!
