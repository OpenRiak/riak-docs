&gt;&gt;&gt; On Jan 30, 2014, at 1:48 PM, Jason Campbell  wrote:
&gt;&gt;&gt;
&gt;&gt;&gt; I'm not sure Riak is the best fit for this. Riak is great for
&gt;&gt;&gt; applications where it is the source of data, and has very strong
&gt;&gt;&gt; consistency when used in this way. You are using it as a cache, where Riak
&gt;&gt;&gt; will be significantly slower than other cache solutions. Especially since
&gt;&gt;&gt; you say that each worker will have a set of documents it is responsible
&gt;&gt;&gt; for. Something like a local memcache or redis would likely suit this use
&gt;&gt;&gt; case just as well, but do it much faster with less overhead.
&gt;&gt;&gt;
&gt;&gt;&gt; Riak will guarantee 3 writes to disk (by default), where something like
&gt;&gt;&gt; memcache or redis will stay in memory, and if local, won't have network
&gt;&gt;&gt; latency either. In the worst case where a node goes offline, the real data
&gt;&gt;&gt; can be pulled from the backend again, so it isn't a big deal. It will also
&gt;&gt;&gt; simplify your application, because node.js can always request from cache
&gt;&gt;&gt; and not worry about the speed, instead of maintaining it's own cache layer.
&gt;&gt;&gt;
&gt;&gt;&gt; I'm as happy as the next person on this list to see Riak being used for
&gt;&gt;&gt; all sorts of uses, but I believe in the right tool for the right job.
&gt;&gt;&gt; Unless there is something I don't understand, Riak is probably the wrong
&gt;&gt;&gt; tool. It will work, but there is other software that will work much better.
&gt;&gt;&gt;
&gt;&gt;&gt; I hope this helps,
&gt;&gt;&gt; Jason Campbell

&gt;&gt;&gt; I'll try to explain this the best I can, although it's a simples
&gt;&gt;&gt; architecture I'm not describing it in my native language :)
&gt;&gt;&gt;
&gt;&gt;&gt;
&gt;&gt;&gt; I have a set of node.js workers (64 for now) that serve as a
&gt;&gt;&gt; cache/middleware layer for a dozen of php applications. Each worker deals
&gt;&gt;&gt; with a set of documents (it's not a distributed cache system). Each worker
&gt;&gt;&gt; updates the documents in memory, and tags them as dirty (just like OS file
&gt;&gt;&gt; cache), and from time to time (for now, it's a 5 seconds window interval),
&gt;&gt;&gt; a persister module will deal with the persistence of those dirty documents
&gt;&gt;&gt; to riak.
&gt;&gt;&gt; If the document isn't in memory, it will be fetched from riak.
&gt;&gt;&gt;
&gt;&gt;&gt;
&gt;&gt;&gt; If you want document X, you need to ask to the corresponding worker
&gt;&gt;&gt; dealing with it. Two different workers, don't deal with the same document.
&gt;&gt;&gt; That way we can guarantee that there will be no concurrent writes to
&gt;&gt;&gt; riak.
