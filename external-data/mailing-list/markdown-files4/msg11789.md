Hi Sean,

This one really is a bug with 2i pagination on Equals term queries[1].

I've fixed it already[2], I'm very sorry that this one got out into the wild. 
It is something that worked then stopped working at some point in a merge. It 
was a pretty bad oversight that we didn't have a test for this condition to 
catch the regression.

I've added a test[3] to the suite for the issue and verified the patch. I guess 
there will be a 1.4.1 soon.

Apologies, again.

Russell

&gt; On Jul 26, 2013, at 6:43 PM, Russell Brown  wrote:
&gt; 
&gt;&gt; For a work around you could use streaming and pagination.
&gt;&gt; 
&gt;&gt; Request smaller pages of data (i.e. sub 60 seconds worth) and use streaming 
&gt;&gt; to get the results to your client sooner.
&gt;&gt; 
&gt;&gt; In HTTP this would look like
&gt;&gt; 
&gt;&gt; http://127.0.0.1:8098/buckets/mybucket/index/test\_bin/myval?max\_results=10000&stream=true
&gt;&gt; 
&gt;&gt; your results will include a continuation like
&gt;&gt; 
&gt;&gt; {"continuation":"g2gCYgAAFXttAAAABDU0OTk="}
&gt;&gt; 
&gt;&gt; and you can use that to get the next N results. Breaking your query up that 
&gt;&gt; way should duck the timeout.
&gt;&gt; 
&gt;&gt; Furthermore, adding &stream=true will mean the first results is received 
&gt;&gt; very rapidly.
&gt;&gt; 
&gt;&gt; I don't think the Ruby client is up to date for the new 2i features, but you 
&gt;&gt; could monkeypatch as before.
&gt;&gt;
