&gt;&gt; I notice you're using the ".withoutFetch()" option when storing;
&gt;&gt; that's guaranteed to create a sibling if you have allow\_multi=true set
&gt;&gt; in the bucket. If that's the case then that behavior is expected; both
&gt;&gt; versions are stored in Riak.
&gt;&gt;
&gt;&gt; Also worth noting is that if you're recently deleted something
&gt;&gt; (explicitly via a delete operation) it's very likely to get a
&gt;&gt; tombstone pass to map/reduce. If you're doing explicit deletes from
&gt;&gt; Riak you need to check the object metadata for the
&gt;&gt; &lt;&lt;"X-Riak-Deleted"&gt;&gt; header being true, and then ignore that object in
&gt;&gt; your map function.

&gt;&gt;&gt; I'm somewhat confused by your question; map/reduce doesn't really have
&gt;&gt;&gt; anything to do with your Java POJO/class.
&gt;&gt;&gt;
&gt;&gt;&gt; When using the Riak Java client and storing a POJO, the default
&gt;&gt;&gt; converter (JSONConverter) uses the Jackson JSON library and converts
&gt;&gt;&gt; the instance of your POJO into a JSON string and stores it in Riak.
&gt;&gt;&gt;
&gt;&gt;&gt; If you change that POJO class and store more things, the resulting
&gt;&gt;&gt; JSON is obviously going to be different (in your case having an
&gt;&gt;&gt; additional field named "minty").
&gt;&gt;&gt;
&gt;&gt;&gt; When doing Map/Reduce, whatever JavaScript or Erlang functions you
&gt;&gt;&gt; provide are executing in Riak and being given the data stored in Riak
&gt;&gt;&gt; (the JSON you stored); they have no connection to Java.
&gt;&gt;&gt;
&gt;&gt;&gt; Can you expand on "Now the map reduce fails for that the new
&gt;&gt;&gt; property" with what exactly the problem is? It sounds like you have a
&gt;&gt;&gt; problem with your JavaScript or Erlang function(s).
