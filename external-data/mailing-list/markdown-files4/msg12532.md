Daniel -

Unfortunately returning the body from a store operation may not
reflect all the replicas (and in the case of a concurrent write on two
different nodes "may not" really means "probably doesn't").

If you do a subsequent fetch after sending both your writes you'll get
back a single vclock with siblings.

Thanks,
- Roach

&gt;&gt; &gt;&gt;
&gt;&gt; &gt;&gt; You should always use vclocks when possible, which in the case it is.
&gt;&gt; &gt;&gt; There are additional issues around doing the delete without a vclock
&gt;&gt; &gt;&gt; and if there's concurrently a store operation occurring.

&gt;&gt; &gt;&gt; Ideally you should look at why you're getting that tombstone sibling.
&gt;&gt; &gt;&gt; If it's simply a case of high write concurrency and you're using
&gt;&gt; &gt;&gt; vclocks with your writes, then there's not much you can do except
&gt;&gt; &gt;&gt; resolve it later (without changing how you're using the DB)... but
&gt;&gt; &gt;&gt; usually these things are caused by writes without a vclock.
