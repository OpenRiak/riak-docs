&gt;&gt;&gt;
&gt;&gt;&gt; \*\*\*\*
&gt;&gt;&gt;
&gt;&gt;&gt; They're both pretty crappy in terms of performance - they read all data
&gt;&gt;&gt; off of disk. If you're using LevelDB you can use the $keys index to pull
&gt;&gt;&gt; back just the keys that in a single bucket.\*\*\*\*
&gt;&gt;&gt;
&gt;&gt;&gt; \*\*\*\*
&gt;&gt;&gt;
&gt;&gt;&gt; A better approach is to maintain a separate bucket - e.g. DocumentCount
&gt;&gt;&gt; - that is used for counting documents. Unfortunately, you can't guarantee
&gt;&gt;&gt; transactional consistency around counts in Riak today, so you'll want to
&gt;&gt;&gt; move maintaining the counts out of Riak and into something else. If you
&gt;&gt;&gt; search the list archives [1], you'll find that Redis has been mentioned as
&gt;&gt;&gt; a good way to solve this problem - counters are stored in Redis and flushed
&gt;&gt;&gt; to Riak on a regular schedule. Because of the lack of consistency
&gt;&gt;&gt; (especially around MapReduce operations), Riak isn't the best choice if you
&gt;&gt;&gt; require counters/aggregations to be stored in the database.\*\*\*\*
&gt;&gt;&gt;
&gt;&gt;&gt; \*\*\*\*
&gt;&gt;&gt;
&gt;&gt;&gt; Once CRDTs [2] make it into mainstream Riak, you can make use of those
&gt;&gt;&gt; data structures to implement distributed counters in Riak.\*\*\*\*
&gt;&gt;&gt;
&gt;&gt;&gt; \*\*\*\*
