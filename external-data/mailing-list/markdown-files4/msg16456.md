&gt;&gt;&gt;&gt;&gt;&gt; Sujay,
&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt; You're right - the best way to verify the backup is to bring up a
&gt;&gt;&gt;&gt;&gt;&gt; separate 5 node cluster, and restore it from the backup files.
&gt;&gt;&gt;&gt;&gt;&gt; The procedure is slightly more involved than untar-ing, though. The
&gt;&gt;&gt;&gt;&gt;&gt; backed up ring directories from the original cluster will contain the 
&gt;&gt;&gt;&gt;&gt;&gt; node
&gt;&gt;&gt;&gt;&gt;&gt; ids (which rely on their IP addresses, etc). Since the new re-hydrated
&gt;&gt;&gt;&gt;&gt;&gt; cluster is likely to have different IPs from the original one, there's a
&gt;&gt;&gt;&gt;&gt;&gt; few more steps you need to take.
&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt; The procedure of standing up a new cluster from backups is outlined
&gt;&gt;&gt;&gt;&gt;&gt; here:
&gt;&gt;&gt;&gt;&gt;&gt; http://docs.basho.com/riak/latest/ops/running/nodes/renaming/#Clusters-from-Backups
&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt; There is one other important step to remember. If, by any chance,
&gt;&gt;&gt;&gt;&gt;&gt; you're bringing up the new cluster on the same network as the old cluster
&gt;&gt;&gt;&gt;&gt;&gt; is running on, be sure to modify the Erlang cookie in the new cluster (so
&gt;&gt;&gt;&gt;&gt;&gt; that as far as Erlang is concerned, they're existing on different 
&gt;&gt;&gt;&gt;&gt;&gt; networks).
&gt;&gt;&gt;&gt;&gt;&gt; The Erlang cookie must be modified in /etc/riak/riak.conf so the new
&gt;&gt;&gt;&gt;&gt;&gt; cluster does not conflict with any existing cluster.
&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt; Let us know if you have any further questions.
&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt; Dmitri

&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; Hi Sujay,
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; The latest release in the 2.0 series is 2.0.6 [0]. Please use this
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; version if you upgrade to 2.0.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; Please also review the documentation about the new 'riak.conf'
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; configuration file [1][2]. 2.x installations should use the new 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; format, but
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; you can continue to use the 'app.config' format from Riak 1.x. To 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; maintain
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; complete backwards compatibility when using 'app.config', please add
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; [{default\_bucket\_props,
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; [{allow\_mult,false}, %% have Riak resolve conflicts and do not 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; return siblings
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; {dvv\_enabled,false}]}, %% use vector clocks for conflict resolution
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; %% other settings
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; ]
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; to 'app.config'. This will ensure that your existing application
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; continues to work exactly as before. When using 'riak.conf', these 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; settings
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; will be applied automatically.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
