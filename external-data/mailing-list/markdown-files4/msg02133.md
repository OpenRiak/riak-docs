&gt;&gt; Hi
&gt;&gt;
&gt;&gt; let me clarify the situation somewhat. The problem is not intermittent
&gt;&gt; in the sense that it occurs randomly. It depends mainly on the size of
&gt;&gt; the answer that riak sends. If the answer is bigger than the network MTU
&gt;&gt; the error will occur most of the time. If its smaller it occurs almost
&gt;&gt; never. The recv operation on the socket will usually return after every
&gt;&gt; received network packet, which means if you call recv fast enough it
&gt;&gt; will never return more bytes than the MTU of the conection.
&gt;&gt;
&gt;&gt; For get operations and put operations where return\_body is True (the
&gt;&gt; default) this obviously depends on the size of the object one is
&gt;&gt; requesting/putting. So it seems to be itermittent if most objects are
&gt;&gt; smaller than about 1400 bytes (most LANs have an MTU of 1492), but some
&gt;&gt; are larger or multiple versions are returned for some of them.
&gt;&gt;
&gt;&gt; The fix I proposed in the bug report (removing the three line) is the
&gt;&gt; correct one. The while loop only terminates if the desired number of
&gt;&gt; bytes have been read or a socket error like a timeout or a closed
&gt;&gt; connection produces an exception.
&gt;&gt;
&gt;&gt; BTW, I have also observed short write errors when storing large objects
&gt;&gt; (larger than the default TCP send buffer I would guess, which is 16k at
&gt;&gt; least on Linux). The fix would be to also loop in snd\_pkt() until all
&gt;&gt; data has been sent.
&gt;&gt;
&gt;&gt; Cheers,
&gt;&gt; Nico
