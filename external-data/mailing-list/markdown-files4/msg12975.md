&gt;&gt; bucket sizes are very predictable due to the nature of the data. I
&gt;&gt; originally used the get/update/set cycle, but as I approach the end of the
&gt;&gt; interval, reading and writing 1MB+ objects at a high frequency kills
&gt;&gt; network bandwidth. So now, I append siblings, and I have a cron that merges
&gt;&gt; the previous siblings (a simple set union works for me, only entire objects
&gt;&gt; are ever deleted).

&gt;&gt; some other method of knowing how much data is in one, I don't see size

&gt;&gt; object is without fetching it, which shouldn't be off by more than a few
&gt;&gt; siblings unless there is a network partition.
&gt;&gt; &gt;&gt;&gt;
&gt;&gt; &gt;&gt;&gt; So aside from size issues, which can be roughly predicted or worked
&gt;&gt; around, is there any reason to not create hundreds or thousands of siblings
&gt;&gt; and resolve them later? I realise sets could work well for my use case, but
&gt;&gt; they seem overkill for simple append operations when I don't need delete
&gt;&gt; functionality. Creating your own CRDTs are trivial if you never need to
&gt;&gt; delete.

&gt;&gt; &gt;&gt;&gt; Forcing siblings other than for testing purposes is not typically a
&gt;&gt; good idea; as you indicate, the object size can easily become a problem as
&gt;&gt; all siblings will live inside the same Riak value.
&gt;&gt; &gt;&gt;&gt;
&gt;&gt; &gt;&gt;&gt; Your counter-example sounds a lot like a use case for server-side
&gt;&gt; CRDTs; data structures that allow the application to add values without
&gt;&gt; retrieving the server-side content first, and siblings are resolved by Riak.
&gt;&gt; &gt;&gt;&gt;
&gt;&gt; &gt;&gt;&gt; These will arrive with Riak 2.0; see
&gt;&gt; https://gist.github.com/russelldb/f92f44bdfb619e089a4d for an overview.
