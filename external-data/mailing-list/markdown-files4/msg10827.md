Riak has a virtual node on physical node structure. So all of your
virtual nodes are running on a single machine. If you were to add
more nodes, some of them would migrate to those new nodes,
distributing storage and load around the cluster.

&gt;&gt; On Wed, Apr 10, 2013 at 6:22 AM, Matthew Von-Maszewski

&gt;&gt;&gt;
&gt;&gt;&gt; Greetings Ben,
&gt;&gt;&gt;
&gt;&gt;&gt; Also, leveldb stores data in "levels". The very first storage level and
&gt;&gt;&gt; the runtime data recovery log are not compressed.
&gt;&gt;&gt;
&gt;&gt;&gt; That said, I agree with Tom that you are most likely seeing Riak store 3
&gt;&gt;&gt; copies of your data versus only one for mongodb. It is possible to dumb
&gt;&gt;&gt; down Riak so that it is closer to mongodb:
&gt;&gt;&gt;
&gt;&gt;&gt; 1. in app.config, look for the riak\_core options, add the following
&gt;&gt;&gt; line:
&gt;&gt;&gt;
&gt;&gt;&gt; {default\_bucket\_props, [{n\_val,1}]},
&gt;&gt;&gt;
&gt;&gt;&gt; This will default the system to only storing one copy of your data.
&gt;&gt;&gt;
&gt;&gt;&gt;
&gt;&gt;&gt; 2. if you are using Riak 1.3, again in app.config, look for the riak\_kv
&gt;&gt;&gt; options:
&gt;&gt;&gt;
&gt;&gt;&gt; change this
&gt;&gt;&gt;
&gt;&gt;&gt; {anti\_entropy, {on, []}},
&gt;&gt;&gt;
&gt;&gt;&gt; to
&gt;&gt;&gt;
&gt;&gt;&gt; {anti\_entropy, {off, []}},
&gt;&gt;&gt;
&gt;&gt;&gt; This will disable Riak's automatic detection and correction of data loss
&gt;&gt;&gt; / corruption. The feature requires an added 1 to 2% data on disk.
&gt;&gt;&gt;
&gt;&gt;&gt;
&gt;&gt;&gt; Matthew

&gt;&gt;&gt;
&gt;&gt;&gt; How many nodes of Riak are you running vs how many nodes of Mongo?
&gt;&gt;&gt;
&gt;&gt;&gt; How much more disk space did Riak take?
&gt;&gt;&gt;
&gt;&gt;&gt; Riak is designed to run as a cluster of several nodes, utilizing
&gt;&gt;&gt; replication to provide resiliency and high-availability during partial
&gt;&gt;&gt; failure. By default Riak stores three replicas of every object you persist.
&gt;&gt;&gt; If you are only running a single node of Riak for your testing purposes, I
&gt;&gt;&gt; suspect this may explain the significant divergence you're seeing when
&gt;&gt;&gt; compared to the disk space used vs a single mongo, as each replica in Riak
&gt;&gt;&gt; is being stored to the same disk.
&gt;&gt;&gt;
&gt;&gt;&gt; Also, Snappy is optimizes for speed over disk utility, which will have a
&gt;&gt;&gt; negligible impact on total disk usage when compared to other compression
&gt;&gt;&gt; libraries such as zlib, etc. That said, for sufficiently large JSON files I
&gt;&gt;&gt; know that BSON's prefixes can add significant overhead to object sizes such
&gt;&gt;&gt; that BSON is actually heavier than the JSON it represents. What is the
&gt;&gt;&gt; average size of the documents you're seeking to store?
&gt;&gt;&gt;
&gt;&gt;&gt; Could you tell us a bit more about what you're trying to achieve with
&gt;&gt;&gt; both Riak and Mongo, respectfully?
&gt;&gt;&gt;
&gt;&gt;&gt; Tom
