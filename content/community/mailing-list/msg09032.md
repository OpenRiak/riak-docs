&gt;&gt; I know you want to follow a criteria for sorting messages, you could
&gt;&gt; stamp each message up to the second precision (storing milliseconds would
&gt;&gt; be a problem) and index them by day (YYYYMMDD), so that way you can find by
&gt;&gt; day using 2i search, and sort by time, so the ID won't be needed for
&gt;&gt; sorting, that way your design will be more loose, no need for
&gt;&gt; synchronization of any type.
&gt;&gt;
&gt;&gt; Now if you really need a sequence, if you feel compelled to use a
&gt;&gt; sequence, just use a dummy database sequence, any PostgreSQL or MySQL
&gt;&gt; service can do that for you, but if you want your app to be loosely couple
&gt;&gt; and just exploit the Riak cluster as much as possible, UUID will be your
&gt;&gt; winning ticket for message ID. If you are unsure UUID will be unique, add
&gt;&gt; another criteria and prefix the UUID with something like sender, etc.
&gt;&gt;
