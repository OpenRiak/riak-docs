&gt;&gt; http://code.google.com/apis/protocolbuffers/docs/techniques.html
&gt;&gt;
&gt;&gt; --snip
&gt;&gt; If you want to write multiple messages to a single file or stream, it is
&gt;&gt; up to you to keep track of where one message ends and the next begins. The
&gt;&gt; Protocol Buffer wire format is not self-delimiting, so protocol buffer
&gt;&gt; parsers cannot determine where a message ends on their own. The easiest way
&gt;&gt; to solve this problem is to write the size of each message before you write
&gt;&gt; the message itself. When you read the messages back in, you read the size,
&gt;&gt; then read the bytes into a separate buffer, then parse from that buffer. (If
&gt;&gt; you want to avoid copying bytes to a separate buffer, check out the
&gt;&gt; CodedInputStream class (in both C++ and Java) which can be told to limit
&gt;&gt; reads to a certain number of bytes.)
&gt;&gt;
&gt;&gt; --end snip
&gt;&gt;
&gt;&gt; You do have to parse a message in its entirety; there is no incremental
&gt;&gt; parsing. If you have a message which wraps 10,000 messages you'll have to
&gt;&gt; load the whole thing in before you can parse.

&gt;&gt;&gt;&gt; Because it's being confused in this conversation, I think it merits
&gt;&gt;&gt;&gt; clarification -- the "protocol" that is used to talk to Riak and Google's
&gt;&gt;&gt;&gt; Protocol Buffers are NOT the same thing. Riak uses a simple length- and
&gt;&gt;&gt;&gt; message-code-prefixed binary protocol, in which the complex messages (ones
&gt;&gt;&gt;&gt; that have bodies and not just the message code) are serialized via Google's
&gt;&gt;&gt;&gt; Protocol Buffers. So, while we don't use the RPC facilities in Google's
&gt;&gt;&gt;&gt; library, the \\*serialization format\\* DOES use Protocol Buffers.
