&gt; that's in the data section (which is your BERT encoded data), the first 
&gt; character in that list is 131, which is not a valid UTF-8 character, and JSON 
&gt; only allows valid UTF-8 characters. With a binary-encoded format, there's 
&gt; always a chance for a control character like that to blow up the JSON 
&gt; generated before and after the MapReduce code is executed. With JSON, content 
&gt; agnosticism only goes as far as the set of legal characters allows.

&gt; On a side note, if the data were a valid representation of a string, you 
&gt; would see it as a string in the log file as well, not just as a list of

&gt;&gt; &gt; &gt; &gt; Andrew,
&gt;&gt; &gt; &gt; &gt;
&gt;&gt; &gt; &gt; &gt; you're indeed hitting a JSON encoding problem here. BERT is binary 
&gt;&gt; &gt; &gt; &gt; data,
&gt;&gt; &gt; &gt; &gt; and won't make the JSON parser happy when trying to deserialize it, 
&gt;&gt; &gt; &gt; &gt; before
&gt;&gt; &gt; &gt; &gt; handing it into the map phase. You have two options here, and none of 
&gt;&gt; &gt; &gt; &gt; them
&gt;&gt; &gt; &gt; &gt; will involve JavaScript as the MapReduce language.
&gt;&gt; &gt; &gt; &gt;
&gt;&gt; &gt; &gt; &gt; 1.) Use the Protobuff API, use Erlang functions to return the value or
&gt;&gt; &gt; &gt; &gt; object (e.g. riak\\_mapreduce:map\\_object\\_value or
&gt;&gt; &gt; &gt; &gt; riak\\_kv\\_mapreduce:map\\_identity), and then run MapReduce queries with 
&gt;&gt; &gt; &gt; &gt; the
&gt;&gt; &gt; &gt; &gt; content type 'application/x-erlang-binary'. However, you're 
&gt;&gt; &gt; &gt; &gt; constrained by
&gt;&gt; &gt; &gt; &gt; client libraries here, e.g. Ruby and Python don't support this content 
&gt;&gt; &gt; &gt; &gt; type
&gt;&gt; &gt; &gt; &gt; for MapReduce on the Protobuffs interface yet, so you'd either 
&gt;&gt; &gt; &gt; &gt; implement
&gt;&gt; &gt; &gt; &gt; something custom, or resort to a client that does, riak-erlang-client 
&gt;&gt; &gt; &gt; &gt; comes
&gt;&gt; &gt; &gt; &gt; to mind, though it was proven to be possible using the Java client too,
&gt;&gt; &gt; &gt; &gt; thanks to Russell. See [1] and [2]
&gt;&gt; &gt; &gt; &gt;
&gt;&gt; &gt; &gt; &gt; 2.) Convert the result from BERT into a JSON-parseable structure inside
&gt;&gt; &gt; &gt; &gt; an Erlang map function, before it's returned to the client.
&gt;&gt; &gt; &gt; &gt;
&gt;&gt; &gt; &gt; &gt; The second approach certainly is less restrictive in terms of API 
&gt;&gt; &gt; &gt; &gt; usage,
&gt;&gt; &gt; &gt; &gt; but certainly involves some overhead inside of the MapReduce request 
&gt;&gt; &gt; &gt; &gt; itself,
&gt;&gt; &gt; &gt; &gt; but is less prone to encoding/decoding issues with JSON.
